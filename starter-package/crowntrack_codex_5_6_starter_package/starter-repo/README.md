# CrownTrack ADV

CrownTrack is an offline-first Android/iOS adventure-bike application in staged development. Stage 3B adds strict signed-v2 CrewLink membership and development-only location-message flows. Signatures prove origin and integrity; they do not encrypt location content or prove physical truth.

Not yet implemented: end-to-end encryption, production accounts/cloud authentication, real GPS/background tracking, production networking, Bluetooth, Nearby Connections, Meshtastic, MapLibre, GPX workflows, or production map data.

## Setup

```powershell
npm install
npm run typecheck
npm run lint
npm test
npm run test:relay
npx expo-doctor
npx expo install --check
```

## Run the mobile shell

```powershell
npm run start
```

The tabs are Map, Packs, Plan, Crew, and SOS. Native Android/iOS builds use Expo SQLite; web intentionally uses the in-memory repository.

## Stage 3B signed CrewLink

- Identity is created only after **Create identity**.
- The Ed25519 private seed is generated by Expo Crypto and stored only in Expo SecureStore.
- SQLite stores public identity, trust, revocation, and pairing replay metadata.
- Validation never creates trust. Riders compare the displayed code and explicitly confirm.
- Missing private keys are never regenerated silently.
- Pairing/trust is distinct from group membership. Owner grants advance epochs; revocation creates a terminal tombstone for that identity in Stage 3B.
- Local outbound updates use a durable outbox and clear only after a verified ACK. Separately, a simulated development rider can deliver a signed inbound update that is verified and stored locally before the Map displays its safe freshness/source state.
- Consent is separate from sharing; policy changes, consent revocation, or authorization loss purge unauthorized queued locations.
- Reset and complete deletion clear local identity/trust/pairing and scoped CrewLink state. Recipient devices and backups may retain data previously delivered.
- Cloud, nearby, and mesh are simulated delivery pipes; they are untrusted and do not make a position current. The in-process rider is not a physical phone.

Stage 3B Android runtime acceptance is currently blocked by the local Windows capture layer, despite an available `emulator-5554`; see `docs/audits/STAGE_3B_ANDROID_NATIVE_QA.md`. iOS runtime is unverified.

## Native Android development

`apps/mobile/android/` is generated Expo prebuild output and is not source-controlled. On Windows, the accepted native build uses the short `X:` project mapping and external Gradle caches to avoid CMake path-length/cache-lock problems. The installed development APK can use Metro on port 8081 for TypeScript-only updates.

Never commit generated native projects, APKs, database snapshots, logs, debug keystores, raw invitations, or private material.

## Architecture

- `apps/mobile`: screen composition and native adapters.
- `packages/crew-identity`: portable identity, Ed25519, canonical pairing, and platform-neutral lifecycle logic.
- `packages/crew-protocol`: versioned CrewLink wire schemas.
- `packages/crewlink`: canonical repository, migrations, simulators, retry, and coordination.
- `packages/core`, `packages/offline-packs`, `packages/ui`, and `packages/gpx`: existing domain/UI contracts.
- `apps/relay`: loopback-only development relay with no production authentication.

Unknown source/access states remain unknown. A route line never implies riding permission.
